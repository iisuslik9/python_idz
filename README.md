## 📋 Отчет по выполнению требований

### 1. Бизнес-логика контроля вместимости складов
В соответствии с правилами логистики, контроль лимита вместимости склада (`capacity`) реализован в слое бизнес-логики (`LogisticsService`). При попытке зарегистрировать новый груз или переместить старый, приложение обращается к СУБД через метод `get_warehouse_current_weight()`. 

База данных на своей стороне выполняет быструю агрегацию `SUM(weight)` **исключительно по тем грузам, которые имеют статус "На складе"**. Вес транзитных (`В пути`) или завершенных (`Доставлен`) грузов не учитывается, что экономит оперативную память приложения и обеспечивает точный учет физического пространства.

### 2. Краткое сравнение: SQL vs ORM


| Критерий сравнения | Подход 1: Чистый SQL (DB-API 2.0) | Подход 2: ORM (SQLAlchemy) |
| :--- | :--- | :--- |
| **Скорость разработки** | **Ниже**. Требуется ручное написание длинных текстовых строк SQL-запросов и маппинг строк в словари. | **Выше**. Вставка, обновление и удаление сущностей происходят через работу с объектами языка Python. |
| **Контроль и оптимизация**| **Максимальный**. Разработчик полностью контролирует синтаксис запроса, индексы и планы выполнения. | **Скрытый**. SQL-код генерируется автоматически. Сложные отчеты требуют изучения громоздкого синтаксиса ORM. |
| **Переносимость кода** | **Низкая**. При смене вендора СУБД (например, с PostgreSQL на MS SQL) потребуется переписывать диалекты SQL. | **Высокая**. Достаточно сменить строку подключения. SQLAlchemy сама переведет вызовы на целевой диалект SQL. |
| **Безопасность инъекций** | Требует явного контроля со стороны разработчика (передача кортежей параметров через оператор `%s`). | Высокая по умолчанию. Все передаваемые параметры экранируются автоматически механизмами библиотеки. |

**Вывод:** Реализованный в проекте паттерн **Repository** совместно с инверсией зависимостей позволил полностью изолировать интерфейс и логику от выбранного режима. Для стандартных CRUD-операций эффективнее использовать ORM, тогда как для кастомных аналитических отчетов со сложными группировками (подсчет доставок водителей) чистый SQL предоставляет более прозрачный контроль над транзакциями.


# Сравнение SQL и ORM способов доступа к БД

## Общая идея

В проекте реализованы два способа работы с базой данных: через прямые SQL-запросы с использованием Python DB-API 2.0 и через ORM SQLAlchemy. Оба подхода выполняют одинаковые операции, но отличаются уровнем абстракции, удобством разработки и контролем над запросами [web:16][web:40].

## SQL-способ

SQL-режим использует `psycopg2`, то есть прямую работу с курсором, вручную написанные запросы и параметризацию через `execute(..., params)`. Такой подход даёт полный контроль над SQL, удобно подходит для точной настройки запросов и хорошо показывает, как реально работает база данных [web:16][web:45].

### Плюсы
- Полный контроль над запросами.
- Проще понять, что именно выполняется в PostgreSQL.
- Удобно для отладки сложных SQL-операций.
- Хорошо подходит для изучения принципов работы DB-API 2.0 [web:16].

### Минусы
- Больше кода.
- Выше вероятность ошибок в синтаксисе SQL.
- Нужно вручную писать CRUD-операции.
- Сложнее сопровождать при росте проекта [web:40].

## ORM-способ

ORM-режим использует SQLAlchemy и работает с объектами Python вместо ручного написания SQL для каждой операции. Это упрощает код, делает бизнес-логику более читаемой и уменьшает количество шаблонных действий при работе с данными [web:40][web:44].

### Плюсы
- Меньше ручного SQL-кода.
- Код ближе к объектной модели предметной области.
- Проще поддерживать и расширять.
- Удобно работать со связями между таблицами [web:40].

### Минусы
- Меньше контроля над итоговым SQL.
- Иногда сложнее понять, что именно выполняется на уровне БД.
- Возможны накладные расходы по сравнению с ручным SQL.
- Для сложных запросов ORM всё равно может требовать перехода к SQL [web:40][web:43].

## Сравнение по критериям

| Критерий | SQL | ORM |
|---|---|---|
| Уровень абстракции | Низкий | Высокий |
| Количество кода | Больше | Меньше |
| Контроль над запросами | Максимальный | Ограниченный |
| Скорость разработки | Ниже | Выше |
| Поддержка связей | Ручная | Встроенная |
| Отладка SQL | Прямая | Через генерацию ORM-запросов |
| Подходит для обучения | Отлично для понимания БД | Отлично для понимания архитектуры |

## Что лучше для этой работы

Для учебного проекта лучше показать оба подхода, потому что это наглядно демонстрирует разницу между низкоуровневым доступом к БД и объектно-ориентированным подходом. Если цель — понять, как формируются запросы и как работает PostgreSQL, полезнее SQL; если цель — сделать код компактнее и удобнее в сопровождении, лучше ORM [web:16][web:40].

## Вывод

SQL-способ даёт больше контроля и лучше подходит для демонстрации базы данных “изнутри”, а ORM-способ быстрее в разработке и удобнее для расширения приложения. Для отчёта можно написать, что оба подхода были реализованы и сравнение показало: SQL выигрывает в прозрачности, ORM — в удобстве и читаемости [web:40][web:43].
